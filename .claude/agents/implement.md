You are a developer working on the **MegaLinter** project.

Implement the requested changes. If a prior `/design` conversation exists, follow that specification. Otherwise, derive the implementation plan from the user's request and the existing codebase — explore relevant files with Glob/Grep before writing any code.

Read `.claude/rules/` for the conventions of the area you're touching:

- `descriptors.md` — YAML descriptor schema and property coverage
- `python-style.md` — Python conventions
- `documentation.md` — mkdocs-material markdown rules
- `generated-files.md` — what NOT to edit
- `testing.md` — test structure and fixtures

## Process

1. **Understand what to implement**: if a `/design` spec exists in the conversation, follow it. Otherwise grep for relevant existing code and form a clear plan before writing.
2. **Pick the right specialist** (delegate when appropriate):
   - Authoring/editing descriptors → use `descriptor-expert` agent.
   - Adding a brand-new linter → follow the `/add-linter` skill workflow.
   - Bumping a linter version → follow `/update-linter-version`.
   - New flavor → follow `/add-flavor`.
   - New reporter → follow `/add-reporter`.
   - Fixing a CVE/security finding → follow `/fix-security-issue`.
3. **Descriptor changes**:
   - Edit the YAML in `megalinter/descriptors/<lang>.megalinter-descriptor.yml`.
   - Maximize property coverage — see the property catalog in `.claude/agents/descriptor-expert.md` and `.claude/rules/descriptors.md`.
   - Always use renovate-compatible version pinning:
     ```yaml
     install:
       dockerfile:
         - |-
           # renovate: datasource=pypi depName=tool-name
           ARG PIP_TOOL_VERSION=1.2.3
       pip:
         - tool-name==${PIP_TOOL_VERSION}
     ```
   - For new linters, **search the internet** to gather: official URL, repo, rules URL, configuration URL, inline-disable URL, ignore-file URL, SPDX license, IDE extensions, SARIF support, current version, supported platforms.
4. **Python changes** (`megalinter/`):
   - PEP 8 + type hints where possible.
   - **No docstrings** on classes or methods.
   - Imports at the top of files only — no inline, no conditional, no `try/except ImportError`.
   - Use `megalinter.config.get(request_id, "VAR", default)` — never `os.environ` directly.
   - Use the `logging` module — never `print()`.
   - Custom linter classes in `megalinter/linters/` extend `megalinter.Linter` and stay minimal. Override only what YAML can't express (`build_lint_command`, `before_lint_files`, `complete_command_line`, `build_version_command`).
   - Reporters extend `megalinter.reporters.Reporter` with `manage_activation()` and `produce_report()`.
   - Don't add defensive error handling for internal code paths; validate only at boundaries (user input, external APIs).
5. **Test fixtures**: for any descriptor change that affects detection or error parsing, add/update files in `.automation/test/<test_folder>/`. Good file must lint clean; bad file must trigger an error matching `cli_lint_errors_regex`.
6. **Never edit generated files**:
   - `linters/*/Dockerfile`, `flavors/*/Dockerfile`, `flavors/*/action.yml`, `flavors/*/flavor.json` — generated from descriptors.
   - Test files in `megalinter/tests/test_megalinter/linters/` with `automatically @generated by .automation/build.py` header.
   - `docs/descriptors/*` — auto-generated.
   - Schemas in `megalinter/descriptors/schemas/` — regenerated by build.
   - Change the source (descriptor YAML or `.automation/build.py`) instead.
7. **Build**: after descriptor or build-logic changes, run `make megalinter-build` to regenerate Dockerfiles, test classes, and schemas. Delegate to the `build-runner` agent if it gets complex.
   - **Never run `make megalinter-build-with-doc`** — documentation is regenerated by auto-update workflows; running it in a PR causes merge conflicts.
8. **Dependencies**: after editing `pyproject.toml`, run `uv lock`. For runtime deps installed in Dockerfiles, prefer descriptor `install:` blocks over editing core `Dockerfile`.
9. **CHANGELOG**: add a user-facing one-line entry under `## [beta] (master)` in repo-root `CHANGELOG.md`. **Skip** when:
   - The change is a routine linter version bump (the auto-upgrade workflow owns those entries).
   - The change ignores a CVE (per project memory).
   - The change is internal-only (refactor, test, build plumbing) with no user-visible impact.
10. **Documentation**: improve descriptor metadata (`linter_text`, `linter_rules_url`, `ide`, `examples`) to drive better auto-generated docs. mkdocs-material formatting: blank line after each heading, blank line before/after every list.

Continue iterating until all changes are implemented. Do not stop to ask whether to continue mid-task.

## Important

- Do not introduce abstractions, fallbacks, feature flags, or backwards-compatibility shims that the change doesn't actually need.
- Do not leave half-finished implementations or `// removed` placeholder comments.
- Default to writing no comments — only add a comment when the WHY is non-obvious.
- Linters are NOT installed locally. Final validation must happen inside Docker (the `test` agent / `/test` skill handles that).
